# 아마존 셀러 카테고리 진입 분석 — 아키텍처 & 파이프라인 설계서

> 작성일: 2026-06-12 · v1.0
> 프로젝트: "전직 아마존 셀러가 데이터로 다시 푸는 카테고리 진입 의사결정"
> 구성: 설계(1~6장) → 비판적 리뷰(7장) → 리뷰 반영 최종 결정사항(8장)

---

## 1. 분석 질문 정의 (모든 설계의 기준점)

**메인 질문**: 신규 셀러 관점에서, 어떤 니치(세부 카테고리)가 진입 가치가 있으며, 어떤 초기 조건(가격대, 차별화 포인트)으로 진입해야 하는가?

서브 질문 4개로 분해:

| # | 서브 질문 | 산출물 |
|---|---|---|
| Q1 | 니치별 경쟁 강도와 수요는 어떻게 다른가? | 니치 기회 스코어카드 |
| Q2 | 저평점 리뷰에 반복되는 미충족 니즈는 무엇인가? | 니치별 unmet needs 리포트 |
| Q3 | 출시 후 1년 내 "안착"하는 상품의 초기 조건은? | 성공 예측 모델 + 피처 해석 |
| Q4 | (선택) 진입 후 가격·경쟁 환경은 어떻게 전개되는가? | Keepa 시계열 사례 분석 |

> 원칙: 모든 분석 단계는 위 4개 질문 중 하나에 매핑되어야 함. 매핑 안 되는 분석은 만들지 않는다(스코프 크리프 방지).

```

                                                            │
Keepa API (선택, Q4) ──────► keepa python client  ──►  data/marts/     (분석용 테이블)
                                                            │
                             ┌──────────────────────────────┘
                             ▼
[변환/분석 Processing]                          [서빙 Serving]

[재현성 Reproducibility]
```

기술 스택 선정 원칙: **분석가 포트폴리오이지 엔지니어링 포트폴리오가 아니다.** Airflow·Spark·클라우드는 의도적으로 배제. 대신 SQL 역량 증명을 위해 DuckDB를 변환 레이어에 명시적으로 사용(채용공고의 SQL 요구 대응).

## 3. 데이터 레이어 설계

### 3.1 소스 및 범위 확정

| 소스 | 용도 | 범위 |
|---|---|---|
| Amazon Reviews 2023 — 리뷰 | Q1, Q2, Q3 | 카테고리 2개로 한정 (아래 기준) |
| Amazon Reviews 2023 — item metadata | Q1, Q3 | 동일 카테고리 |
| Keepa API | Q4 | 최종 후보 니치의 ASIN 100~300개, 1개월 구독 |
| 수기 가정 테이블 | 마진 시뮬레이션 | FBA 수수료율, 물류비 가정 (출처 명시) |

**카테고리 선정 기준**: ① 본인이 셀러 시절 실제 취급했던 카테고리 1개(서사의 핵심) + ② 대조군으로 경쟁 구조가 다른 카테고리 1개. 3개 이상 금지 — 용량과 분석 깊이 모두 망가짐.

**분석 기간**: 2015-01 ~ 2023-09. 1996년부터 받지 않는다(초기 데이터는 희소하고 현재 마켓플레이스 구조와 무관).

### 3.2 저장 구조 (3-레이어)

```
data/
├── raw/        # 소스 그대로. 절대 수정 금지 (parquet, 카테고리/연도 파티션)
├── staging/    # 타입 정리, 중복 제거, 필터링. 1소스=1테이블 원칙
└── marts/      # 분석 단위 테이블 (DuckDB SQL로 생성)
    ├── dim_product        # ASIN 단위: 가격, 브랜드, 니치 라벨, 첫 리뷰일
    ├── fct_review         # 리뷰 단위: 평점, 텍스트, verified 여부
    ├── mart_niche_monthly # 니치×월: 리뷰 수, 신규 상품 수, 평점 분포 → Q1
    └── mart_launch        # 상품×출시 코호트: 초기 조건 + 1년 후 결과 → Q3
```

### 3.3 핵심 정의 (데이터 사전에 명문화)

- **니치(niche)**: 메타데이터 category 트리의 leaf 노드 기준. 단, 상품 수 30개 미만 leaf는 부모로 합침
- **출시일(launch date)**: 메타데이터에 출시일 없음 → **첫 리뷰 날짜로 프록시** (한계 7장에서 검토)
- **수요 프록시**: 월별 신규 리뷰 수 (리뷰≠판매 문제는 7장)
- **성공(Q3 타깃)**: 첫 리뷰 후 12개월 내 누적 리뷰 ≥ N개 AND 평균 평점 ≥ 4.0. N은 니치별 중앙값 기반으로 설정(절대값 고정 금지 — 니치마다 리뷰 볼륨이 다름)

## 4. 파이프라인 단계별 설계

### Stage 0 — 환경/스캐폴딩 (0.5주)
repo 구조, Makefile, requirements, 데이터 사전 템플릿. 노트북은 `01_`, `02_` 번호 체계.

### Stage 1 — Ingestion (1주)
- HuggingFace `datasets` 라이브러리로 카테고리별 스트리밍 다운로드 → 연도별 parquet 저장
- 수집 직후 검증 스크립트: 행 수, null 비율, 기간 커버리지, ASIN 중복률 기록 → `data_quality_report.md` 자동 생성 (데이터 품질 검증 습관 자체가 어필 포인트)

### Stage 2 — Staging 변환 (1주)
- DuckDB SQL: 타입 캐스팅, 가격 파싱($ 문자열 → 숫자, 결측 처리), category 트리 → 니치 라벨 정규화
- 중복 리뷰 제거, unverified 리뷰 플래그
- **여기서 가장 시간이 오래 걸림을 전제로 일정 배분** (실데이터의 메타데이터는 지저분함)

### Stage 3 — Q1: 니치 기회 스코어링 (1.5주)
- mart_niche_monthly 기반 지표: 수요 성장률(리뷰 증가율), 경쟁 집중도(상위 5개 상품 리뷰 점유율, HHI), 신규 진입 속도, 평점 갭(평균 평점이 낮은 니치 = 품질 개선 진입 기회)
- 지표 표준화 후 가중합 → 기회 점수. **가중치는 민감도 분석 필수** (가중치 바꿔도 상위 니치가 유지되는지)
- 본인이 팔았던 니치의 점수를 회고 검증: "당시 이 분석이 있었다면 진입했을까?"

### Stage 4 — Q2: 미충족 니즈 텍스트 마이닝 (1.5주)
- 상위 기회 니치 3~5개의 1~3점 리뷰만 추출
- 임베딩(sentence-transformers) → 클러스터링(HDBSCAN) → 클러스터별 대표 리뷰를 LLM으로 라벨링/요약
- 산출: 니치별 "불만 토픽 × 빈도 × 예시" 테이블 → 차별화 진입 포인트 도출
- 검증: 클러스터 라벨을 본인이 직접 샘플 50건 읽고 정합성 확인 (LLM 출력 맹신 금지)

### Stage 5 — Q3: 안착 예측 모델 (1.5주)
- 단위: mart_launch (출시 코호트)
- 피처: **출시 후 첫 90일 내 정보만** (가격, 니치 경쟁 지표, 초기 평점, 초기 리뷰 텍스트 특성) — 누수 방지의 핵심
- 분할: **시간 기반 split** (예: ~2021 출시분 학습 / 2022 출시분 검증) — 랜덤 split 금지
- 모델: 로지스틱 회귀(베이스라인, 해석) → LightGBM → SHAP으로 피처 기여 해석
- 평가: PR-AUC 중심(클래스 불균형), 그리고 "이 모델로 진입 결정 시 기대 효용" 형태의 비즈니스 해석 한 단락

### Stage 6 — Q4(선택): Keepa 사례 분석 (1주, 예산 시)
- Stage 3 상위 니치의 ASIN 100~300개 가격·BSR 히스토리 수집
- "성공/실패 상품의 진입 후 18개월 가격 전개" 사례 내러티브 2~3개

### Stage 7 — 서빙/패키징 (1.5주)
- Streamlit: 니치 탐색기(필터 + 스코어카드 + 불만 토픽)
- 최종 리포트: **"투자 의사결정 메모" 형식** (요약 → 추천 니치 → 근거 → 리스크 → 한계). 노트북 나열이 아니라 경영진 보고서처럼
- README: 셀러 시절 일화로 시작하는 스토리텔링

**총 일정: 약 8~9주** (주 15~20시간 기준)

## 6. 데이터 수집에서 Playwright의 위치

Playwright MCP로 아마존을 직접 크롤링하는 것은 **배제 권장**. 이유: ① 아마존 ToS 위반 — 포트폴리오 공개 시 면접에서 컴플라이언스 감점 리스크(특히 핀테크), ② 안티봇으로 안정 수집 자체가 어려움, ③ 허깅페이스 데이터로 충분. Playwright를 쓰고 싶다면 ToS상 허용되는 보조 소스(예: 공개 통계 페이지) 정도로 한정하고, 그마저도 핵심 파이프라인에 의존성으로 넣지 말 것.

## 7. 비판적 셀프 리뷰 (Red Team)

설계를 채용 담당자/시니어 분석가 관점에서 공격해본 결과. **심각도 순.**

### R1. 리뷰 ≠ 판매 (심각도: 높음 / 회피 불가, 관리 대상)
리뷰 수를 수요 프록시로 쓰지만 리뷰 전환율은 카테고리·가격대·시기별로 다름(통상 구매의 1~2%). 저가 소모품은 리뷰율이 낮고, 리뷰 요청 정책 변화(2016 인센티브 리뷰 금지 등)로 시계열 단절도 있음.
**대응**: ① limitations.md에 1순위로 명시, ② 니치 "간" 절대 비교가 아니라 동일 니치 "내" 상대 비교와 추세 중심으로 해석 프레임 제한, ③ Q4 Keepa BSR로 일부 니치에서 리뷰-랭킹 상관을 교차 검증. 면접에서 반드시 물어볼 질문이므로 오히려 준비된 답변 기회로 전환.

### R2. 생존 편향 (심각도: 높음)
데이터셋에는 리뷰가 1개라도 달린 상품만 존재. **진입했다가 리뷰 0개로 사라진 상품(진짜 실패)은 보이지 않음.** Q3 "성공 예측"은 실제로는 "리뷰를 받은 상품 중 안착 vs 부진" 예측임.
**대응**: 타깃 정의를 정직하게 재명명("시장 안착" vs "조기 정체"), 리포트에 모집단 정의를 명시. 이걸 숨기면 면접에서 무너지고, 먼저 밝히면 신뢰를 얻음.

### R3. 출시일 프록시 오차 (심각도: 중간)
첫 리뷰일 = 출시일 가정은 리뷰가 늦게 달리는 상품에서 체계적으로 어긋남. 특히 Q3의 "첫 90일 피처"가 실제로는 출시 6개월 후일 수 있음.
**대응**: 메타데이터에 date 필드가 있는 상품 서브셋으로 프록시 오차 분포를 직접 측정해 부록에 수록. 오차가 크면 "첫 리뷰 후 90일"로 프레임 자체를 정직하게 변경.

### R4. Q3 데이터 누수 위험 (심각도: 중간 — 설계로 차단됨, 실행 규율 필요)
"12개월 내 리뷰 N개" 타깃에 "초기 리뷰 증가 속도" 피처를 넣으면 타깃을 피처로 예측하는 자기참조. 또한 니치 경쟁 지표를 전체 기간으로 계산하면 미래 정보 누수.
**대응**: 모든 피처는 "출시 시점에 알 수 있었는가" 테스트 통과 필수. 니치 지표는 출시 시점까지의 데이터로만 계산(point-in-time join). 코드 리뷰 체크리스트화.

### R5. 2023년 9월 컷오프 (심각도: 중간)
"2026년에 2023년 데이터로 진입 추천?"이라는 공격 가능. 특히 아마존은 2024~25년 중국 직발송(Temu류) 경쟁으로 구조가 변함.
**대응**: 리포트 프레이밍을 "지금 이 니치에 들어가라"가 아니라 **"진입 의사결정 프레임워크 구축 + 과거 데이터 검증"**으로 설정. Q4 Keepa(현재 데이터)가 최신성 보강. 이 프레이밍이면 컷오프는 약점이 아니라 범위 설정이 됨.

### R6. 니치 = category leaf 가정 (심각도: 중간)
아마존 카테고리 트리는 지저분함(중복 분류, 판매자 임의 등록, 깊이 불균일). leaf 기준 니치가 실제 시장 단위와 어긋날 수 있음.
**대응**: Stage 2에서 니치 라벨 품질 검사(니치당 상품 수 분포, 수동 샘플 검수 30개). 필요시 제목 임베딩 클러스터링으로 니치 재정의 — 단, 이건 일정 +1주 리스크이므로 품질이 심각할 때만.

### R7. 일정 낙관 편향 (심각도: 중간)
8~9주는 모든 단계가 순항할 때 기준. 경험상 Stage 2(정제)와 Stage 4(LLM 라벨링 검수)가 1.5~2배 걸림.
**대응**: **MVP 컷 정의** — Q1+Q2+리포트만으로도 제출 가능한 포트폴리오가 되도록 단계를 독립적으로 완결. Q3는 확장, Q4는 보너스. "다 못 끝내면 전부 미완성"인 구조 금지.

### R8. 용량/컴퓨팅 (심각도: 낮음~중간)
카테고리에 따라 리뷰 수천만 건 → 로컬 메모리 초과 가능.
**대응**: 스트리밍 다운로드 + 연도 파티션 parquet + DuckDB(out-of-core) 조합이면 노트북에서 처리 가능. 임베딩은 저평점 리뷰 서브셋(수십만 건)에만 적용. 그래도 크면 니치 샘플링.

### R9. "LLM 사용"이 오히려 흔해 보일 위험 (심각도: 낮음)
2026년 기준 "리뷰를 LLM으로 요약"은 더 이상 차별점이 아님.
**대응**: LLM은 도구로만, 차별점은 **검증 설계**(수동 샘플 정합성 확인, 클러스터 안정성)와 **의사결정 연결**에 둠. README에서 LLM을 전면에 내세우지 않음.

### R10. 대시보드 스코프 크리프 (심각도: 낮음)
Streamlit에 기능을 계속 붙이다 분석 깊이가 희생되는 전형적 함정.
**대응**: 화면 3개(니치 스코어카드 / 니치 상세 / 불만 토픽)로 고정. 그 이상 금지.

### 리뷰 총평
설계의 최대 약점은 기술이 아니라 **프록시의 정직성(R1~R3)**이다. 세 가지 모두 회피 불가능하므로, 이 프로젝트의 진짜 차별화는 "한계를 숨기지 않고 정량화해서 의사결정 프레임에 반영하는 태도"가 되어야 한다. 그것이 limitations.md와 의사결정 메모 형식을 설계에 넣은 이유.

## 8. 리뷰 반영 최종 결정사항 요약

1. 리포트 프레이밍: "니치 추천"이 아니라 **"진입 의사결정 프레임워크 + 과거 데이터 검증"** (R1, R5)
2. Q3 타깃을 "성공 예측"이 아니라 **"시장 안착 vs 조기 정체"로 명명** + 모집단 한계 명시 (R2)
3. 출시일 프록시 오차를 **직접 측정해 부록 수록** (R3)
4. 피처는 point-in-time 원칙, 시간 기반 split 강제 (R4)
5. **MVP = Q1+Q2+리포트**, Q3 확장, Q4 보너스로 단계별 완결 구조 (R7)
6. 아마존 직접 크롤링(Playwright 포함) 배제 (6장)
7. 카테고리 2개, 대시보드 화면 3개로 스코프 하드캡 (R6, R10)
