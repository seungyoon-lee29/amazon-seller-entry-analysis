## 2026-06-12 — 1일차: 주제 선정부터 Stage 2 변환까지

### 1. 주제 선정 과정

**출발점**: 이커머스/핀테크 데이터 분석 직무 포트폴리오. 보유 자산은 ① 아마존 셀러 경험
(hexagon wall shelves 판매, 실데이터는 없음) ② 모의투자 대회 수상(기록 없음) ③ 부트캠프
수료(EDA/시각화/ML).

**핵심 판단**: 신입 포트폴리오 대부분이 "남의 데이터로 남의 질문에 답하는" 구조
(Olist, Titanic 등)라서, 차별화 축을 기술이 아니라 **서사**에 둠 —
"셀러 시절 감으로 했던 의사결정을 데이터로 검증한다."

**후보 비교**: A) 아마존 셀러 진입 분석, B) 개인투자자 행동 편향(핀테크), C) FDS 사기 탐지.
→ A 선택. 이유: 본인 경험과 직결(면접 방어력), 공급자 관점 분석은 희소, 전략기획 직무까지 커버.
C는 데이터셋이 너무 흔해 단독 대표 프로젝트로 부적합 판정.

### 2. 데이터 소스 결정 (외부 유료 데이터 제외)

조사 결과 핵심 사실: **Keepa API는 리뷰 텍스트를 제공하지 않는다** (평점·리뷰 수
시계열만). 반면 Amazon Reviews 2023(McAuley Lab)은 리뷰 텍스트+메타데이터를 무료
제공하지만 **가격·랭킹 시계열이 없고 2023-09에 끊긴다.**
→ 결론: 이번 버전은 HF(Amazon Reviews 2023) 데이터만으로 닫는다. Keepa 등 추가 소스는
비용·범위 부담 때문에 이번 버전에서 제외했다.

### 3. 설계 핵심 (상세: 아키텍처_파이프라인_설계.md)

- 분석 질문 3개로 분해: Q1 니치 스코어링 / Q2 미충족 니즈 마이닝 / Q3 안착 예측
- 3레이어 저장(raw 불변 → staging 정제 → marts 분석 단위), DuckDB+Polars, 일정 8~9주
- **레드팀 리뷰에서 나온 가장 중요한 교훈**: 최대 리스크는 기술이 아니라 프록시의 정직성.
  ① 리뷰≠판매 ② 생존 편향(리뷰 0개 상품은 데이터에 없음) ③ 첫 리뷰일≠출시일.
  → 대응: 리포트 프레이밍을 "니치 추천"이 아니라 "의사결정 프레임워크 + 과거 데이터 검증"으로,
  Q3 타깃은 "성공"이 아니라 "시장 안착 vs 조기 정체"로 명명, 한계는 측정해서 부록에 수록.
- MVP 컷: Q1+Q2+리포트만으로도 제출 가능하게 단계별 독립 완결 구조.

### 4. Stage 0~1: 스캐폴딩 + 수집 (실제 작업)

**카테고리 확정**: 메인 Home_and_Kitchen (리뷰 31.4GB / 메타 11.8GB jsonl),
교차등록 보완용 Tools_and_Home_Improvement (메타만, 4.9GB). 별도 대조 카테고리는
"니치 비교는 카테고리 내에서 충분"하므로 스코프 컷 (decisions.md D-002).

**수집 설계의 핵심 아이디어** (D-003): 31GB 리뷰를 통째로 저장하지 않고, 스트림 1회
통과로 이중 출력 — (a) 전체 카테고리 slim(텍스트 제외 5개 컬럼 → 니치 월간 집계용),
(b) 타깃 니치 ASIN만 full(텍스트 포함 → 텍스트 마이닝용). 기간 필터(2015-01~2023-09)도
이 단계에서 적용해 저장량 절감.

**트러블슈팅 ①** — `datasets` 라이브러리가 데이터셋 로드를 거부:
```
RuntimeError: Dataset scripts are no longer supported, but found Amazon-Reviews-2023.py
```
원인: 이 레포는 구식 "로딩 스크립트" 방식인데 datasets>=3.0부터 보안상 차단됨.
해결: `huggingface_hub.HfFileSystem`으로 raw jsonl을 직접 열어 라인 단위 스트리밍
(common.py의 stream_jsonl). 교훈: 라이브러리 추상화가 막히면 한 단계 아래(파일시스템
레벨)로 내려가면 되는 경우가 많다. (D-007)

**스모크 테스트 결과** (메타 5,000행 / 리뷰 20,000행):
- 셸프 니치 매칭 22개, 샘플 육안 검수 결과 오탐 0 (전부 실제 wall/floating shelves)
- 가격 파싱 성공률 51.4% → **가격 결측 처리가 이후 단계의 주요 과제**로 기록됨

### 5. Stage 2: 변환 SQL (staging → marts)

**구조**: `01_staging.sql`(타입 정리·중복 제거) → `02_marts.sql`(분석 단위 테이블)
→ `run_transform.py`(템플릿 치환 후 DuckDB 실행, marts를 parquet으로도 내보냄).

마트 4개:
- `dim_product`: 상품 차원. 첫 리뷰일=출시일 프록시, 30개 미만 leaf는 OTHER_SMALL로 병합
- `fct_review`: 타깃 니치 리뷰(텍스트) — Q2용
- `mart_niche_monthly`: 니치×월 패널 (리뷰 수, 활성 상품 수, 저평점 비율, 신규 진입 수,
  상위5 점유율) — Q1용
- `mart_launch`: 출시 코호트 — Q3용. **누수 방지가 이 테이블의 존재 이유**:
  피처는 첫 90일 정보 + "출시 시점까지의" 니치 상태만(`WHERE s.review_date <
  l.first_review_date`), 결과는 12개월 후. launch_year를 시간 기반 split 키로 포함.
  12개월 관측이 불가능한 코호트(2022-10 이후 출시)는 제외해 우측 절단 방지.

**트러블슈팅 ②** — 0행 출력 시 parquet 파일이 아예 안 생겨 하위 파이프라인이 깨짐:
스모크 샘플에서 타깃 니치 리뷰가 0건이라 reviews_full 파일 미생성 → transform 실패.
해결: ChunkedParquetWriter에 empty_schema 옵션 추가, 0행이어도 스키마 있는 빈 파일 생성.
교훈: "데이터가 없는 경우"도 파이프라인의 정상 경로로 설계해야 한다.

**트러블슈팅 ③** — glob 패턴 버그: `--sample` 모드에서 `meta__sample*`을 찾는 오타성
로직 → `{prefix}*_sample*.parquet`으로 수정. 교훈: 파일명 규약과 glob은 테스트로만 잡힌다.

**트러블슈팅 ④** — DuckDB WAL 삭제 불가 (`Operation not permitted`):
이 작업 환경의 마운트 폴더는 파일 삭제가 보호되어 DuckDB가 WAL을 정리하지 못함.
해결: 삭제 권한 허용 + run_transform에 `--db` 옵션 추가(필요시 /tmp에 DB 생성).
어차피 marts는 parquet으로 내보내므로 DB 파일 위치는 중요하지 않게 설계 변경.
교훈: 환경 제약은 우회 가능하게 경로/출력을 이중화해두면 강건해진다.

**최종 스모크 검증** (전 단계 통과):
```
dim_product 5,000 / stg_review_slim 18,835 / mart_niche_monthly 57 / mart_launch 31
```

## 2026-06-14 — 2일차: 마트 버그 수정 + Stage 3(Q1) 스코어링 착수

### 1. 마트 버그: 가짜 빈 니치 (`''`)

스모크 마트를 다시 들여다보다 발견: `dim_product`에 니치가 빈 문자열인 상품이 398개
(전체의 8%). 원인은 메타데이터의 `categories`가 비어 있는 상품 — leaf 추출
(`regexp_extract`)이 빈 문자열을 반환하고, 그 빈 그룹이 30개 이상이라 OTHER_SMALL로
병합되지 않고 **가짜 니치로 살아남음**. 니치 마트는 `niche != 'OTHER_SMALL'`만
걸러서 이 `''` 니치는 그대로 통과 → 전체 수집 시 니치 스코어링을 오염시킬 시한폭탄.

수정: ① `01_staging.sql`에서 빈 leaf를 `nullif(...,'')`로 NULL화, ② `02_marts.sql`
niche CASE에 `category_leaf IS NOT NULL` 가드 추가 → NULL leaf는 OTHER_SMALL로.
재실행 후 가짜 `''` 니치 0개, 398개가 OTHER_SMALL(3219→3617)로 흡수 확인 (D-008).
교훈: **결측은 별도 버킷으로 보내야지, 우연히 큰 결측 그룹이 "값"처럼 행세하면 안 된다.**

### 2. Stage 3 / Q1: 니치 기회 스코어링 (`src/analyze/niche_score.py`)

설계서 Stage 3을 구현. 지표 4개를 모두 "높을수록 진입 매력적"으로 부호 정렬:
- `demand_growth`  최근 12개월 / 직전 12개월 리뷰 증가율 (수요 모멘텀)
- `market_openness` 1 − 상위5 상품 리뷰 점유율 (집중도 낮을수록 진입 여지)
- `quality_gap`    저평점(≤2★) 비율 (불만 많을수록 품질 개선 진입 기회, Q2 연결)
- `uncrowded`      1 − 신규진입강도(신규/활성) (포화도 낮을수록 매력)

z-표준화 → 가중합. **이 단계의 진짜 산출물은 점수가 아니라 민감도**: 가중치를
Dirichlet(α=8)으로 500회 섭동해 ① 섭동 랭킹 vs 기본 랭킹의 평균 Spearman,
② 각 니치가 상위 N에 드는 빈도(`prob_top_n`)를 보고. "가중치를 흔들어도 상위
니치가 유지되는가"를 정량화 (설계서: "가중치는 민감도 분석 필수").

**L-1(리뷰≠판매) 반영 방식**: 수요를 절대 리뷰량이 아니라 *증가율*로만 점수에
넣어, 니치 간 절대 비교를 구조적으로 회피. 스코어카드 상단에도 해석 제한을 명시.

검증:
- 스모크 데이터는 니치당 활성상품 1개 수준으로 너무 얇아 점수가 퇴화(평균 Spearman
  1.0) → 엔진 동작만 확인용. 실제 의미는 야간 전체 수집 데이터에서 나옴
- 합성 데이터(니치 20개, 지표 분산 부여)로 민감도 로직 별도 검증: 평균 Spearman
  0.91, `prob_top_n`이 1위 니치 0.83 → 6위 0.26으로 깔끔히 변별 → 로직 정상

산출물: `data/marts/mart_niche_score.parquet`(니치별 지표·z·랭크 안정성),
`docs/q1_niche_scorecard.md`(자동 생성 스코어카드, 셀러 타깃 니치 회고 검증 줄 포함).
`make analyze` / `make analyze-sample` 추가, `make smoke`를 transform+analyze까지
확장해 진짜 end-to-end 스모크로 만듦. numpy를 requirements에 명시.

### 4. 수집을 재개 가능(resumable)하게 개조 — 전원 차단 대비

전체 수집을 처음 백그라운드로 돌렸다가 깨달음: 31GB 스트림을 수 시간 받는데 **중간에
전원/네트워크가 끊기면 처음부터**였다(체크포인트 없음). 단일 parquet을 쓰는 구조라
중단되면 footer 없는 손상 파일만 남아 진행분이 전부 날아갔다.

개조(D-010):
- `iter_jsonl_offsets`: HfFileSystem의 바이트 오프셋 seek로 jsonl 재개 스트리밍.
  seek/tell이 HTTP range로 동작하는지 실파일로 먼저 검증(이미 받은 부분 재다운로드 안 함).
- `ShardSink`: 출력을 샤드(part0000…)로 쪼개 쓰고 완료 시 단일 canonical로 병합·정리.
- `save_ckpt`(tmp→fsync→rename, 원자적): 샤드 경계마다 offset·shard_idx·카운터 저장.
- 재실행 안전: canonical 있고 ckpt 없으면 완료로 간주해 skip, ckpt 있으면 그 지점부터 재개.

**검증이 핵심이었다**: 실제 HF 파일로 ① 체크포인트 직후 중단 ② 인터벌 중간 중단(미완 샤드
덮어쓰기) 두 시나리오에서 재개 결과가 무중단 실행과 **byte-identical(중복 0/누락 0)** 임을
확인. 샘플(--limit) 경로도 기존과 동일 수치 재확인.

운영: `scripts/run_full_ingest.sh`(단계별 skip/resume + 리뷰 5회 자동 재시도)를

### 2. Q3 — 안착 예측 모델 (`src/analyze/settlement_model.py`)

**가장 중요한 설계 판단(D-015) — 라벨 정의로 누수 차단**: 피처는 첫 90일 정보인데
라벨을 12개월 누적 리뷰로 잡으면 첫 90일이 라벨에 기계적으로 포함됨(자기참조, 설계서 R4).
→ 라벨을 **초기창 이후(90~365일) 트랙션**으로 정의(`reviews_12m - reviews_90d`).
점검: corr(reviews_90d, 후기 트랙션)=0.73 vs corr(reviews_90d, 12개월 전체)=0.83 —
후기창 정의가 기계적 성분을 0.10 줄임. 남은 0.73은 정당한 조기 신호.

안착 = `later ≥ max(니치 P75, 3) AND avg_rating_12m ≥ 3.5`(니치 상대 임계, config화).
양성률 train 19.5%/valid 13.7%/test 14.3%.
> ※ 이 정의/수치는 *당일(수정 전)* 값이다. 다음날 자체 감사에서 잔여 누수 2건을 발견해
> 평점 게이트 제거 + 니치 임계 train-only로 수정했다 → **5일차 참조**(최종 수치는 거기).

모델: LogReg(중앙값대치+표준화, 베이스라인) → LightGBM(결측 native, scale_pos_weight,
valid early stopping). **시간 split**(train ≤2020 / valid 2021 / test 2022, 랜덤 금지).
불균형이라 **PR-AUC 1차**, 임계값은 valid에서 **MCC 최대화**로 골라 test 적용
(precision/recall/MCC + 혼동행렬 보고).

**결과(test 2022)**: LightGBM PR-AUC 0.603(양성률 0.143 대비 4.2배), ROC-AUC 0.875,
MCC 0.536(베이스라인 0.504 상회). SHAP(beeswarm 저장): 첫 90일 리뷰 수 ≫ 첫 90일 평점 >
가격. 즉 출시 직후 모멘텀·초기 평점 관리가 안착의 핵심 레버 → Q2 미충족 니즈가 초기 평점
